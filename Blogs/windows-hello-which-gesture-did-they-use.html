Reverse Engineering

Which Windows Hello gesture did they actually use to sign in — face, fingerprint, PIN or password?

Microsoft does not ship a single report that answers this. Entra tells you "Windows Hello for Business" and stops there. So I went into the event logs and built one myself. Here is exactly how the script works and where every number comes from.

EW
EndpointWeekly · 29 June 2026 · ~12 min read · Get-TodayLogons.ps1

The problem

You have rolled out Windows Hello for Business on a hybrid cloud trust setup. A user is enrolled for PIN, fingerprint and face. Simple question from your security team:

"Today, when this person signed in — did they use face, fingerprint, PIN, or did they just type their password?"

You open the Entra sign-in logs expecting an answer. You get this:

Entra admin center› Users› Sign-in logs› Authentication Details
Entra ID — Authentication Details
Authentication method: Windows Hello for Business
Authentication method: Windows Hello for Business
Authentication method: Password

"Windows Hello for Business." That's it. Face, fingerprint and PIN all collapse into one label. Intune doesn't break it down either. There is no gesture-level report anywhere in the portal — not in Entra, not in Intune, not in the Authentication methods activity blade.

Gotcha: This isn't an oversight Microsoft forgot to fix. From the identity provider's point of view, face, fingerprint and PIN are genuinely indistinguishable. Keep reading — the reason is the whole point.

Why Microsoft can't show you (the mechanism)

Windows Hello for Business does not store three credentials. It stores one — an asymmetric key pair, protected by the TPM. Your PIN, your fingerprint and your face are not credentials. They are three gestures that all unlock the same private key.

When that key signs the authentication challenge and talks to your DC / Entra, the wire looks identical regardless of which gesture released it. PKINIT with a TPM-backed key is PKINIT with a TPM-backed key. So the cloud, correctly, reports one thing: "Windows Hello for Business."

A password is the odd one out — it never touches that key, which is why Entra can separate "Password" from "WHFB". But it still can't split the three gestures.

Note: This means the answer can only exist locally, on the device — at the moment the gesture unlocks the key, before anything leaves the machine. That's where we have to look.

First, let me confirm what we're dealing with. On a hybrid cloud trust device, dsregcmd /status is the source of truth:

Command Prompt (Administrator)
C:\> dsregcmd /status
AzureAdJoined : YES
DomainJoined : YES
NgcSet : YES
KeyProvider : Microsoft Platform Crypto Provider
TpmProtected : YES
OnPremTgt : YES
CloudTgt : YES

AzureAdJoined: YES + DomainJoined: YES = hybrid joined. OnPremTgt + CloudTgt = cloud trust. NgcSet: YES = Windows Hello is provisioned, TPM-backed. Good — this is the scenario we care about.

Where the data actually lives: three event logs

There is no single "Windows Hello usage" log. The answer is scattered across three logs, and you have to stitch them together. Here they are, with full paths so you can open them yourself in Event Viewer.

What it tells youLog (Applications and Services Logs › Microsoft › Windows › …)
Face / fingerprint was usedBiometrics › Operational
The Hello key was used (PIN or biometric)HelloForBusiness › Operational
A sign-in / unlock happened at allSecurity (Windows Logs)

The enrollment side (what's set up) is even simpler — it's a single registry value, a bitmask under the user's SID:

Registry Editor
HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\WinBio\AccountInfo\<SID>
EnrolledFactors  REG_DWORD  0x0000000A  (10)

EnrolledFactors is a bitmask of the Windows Biometric Framework factor types:

BitFactor
0x02Face (facial features)
0x04Voice
0x08Fingerprint
0x10Iris

Our value is 10 = 0x08 | 0x02 = fingerprint + face enrolled. That's exact, no guessing. But enrollment is only half the question. The hard part is usage — which gesture they actually used. That's where the detective work started.

The detective work: from "everything is PIN" to four clean methods

Clue 1 — Fingerprint is easy. It announces itself.

Open the Biometrics Operational log. Every successful fingerprint match writes an Event ID 1004, and it names the sensor:

ℹ Information — Event ID 1004 — Biometrics/Operational
Message: The Windows Biometric Service successfully
identified [user] using sensor: Goodix MOC Fingerprint
(USB\VID_27C6&PID_6594\...).

So fingerprint = an 1004 event with a fingerprint sensor name. Done. I built the first version of the script on that, plus the Security log for logons, and a simple rule: "if there's a Hello key event near the logon, it's PIN, otherwise password." I gave it a generous ±90 second window to be safe.

That was the mistake.

Watch out: The HelloForBusiness/Operational log is noisy. The Hello key gets touched constantly in the background — SSO, key reads, protector writes (Event 5702) fire dozens of times an hour. With a ±90s window, every sign-in had a Hello event nearby. The script proudly reported everything as PIN. Password sign-ins vanished.

Clue 2 — Get the real anchor: the unlock event

The fix for the noise was to stop guessing from loose timing and anchor on the exact moment of a sign-in. The Security log has two anchors:

  • Event 4624 — an account was logged on (Type 2 = interactive, Type 11 = cached)
  • Event 4801 — the workstation was unlocked

But when I tested it, the unlock events weren't there. The lock screen unlocks I'd just done produced nothing in the Security log. Here's why:

Command Prompt (Administrator)
C:\> auditpol /get /subcategory:"Other Logon/Logoff Events"
System audit policy
Category/Subcategory Setting
Logon/Logoff
Other Logon/Logoff Events No Auditing
Gotcha: Workstation lock/unlock auditing (Events 4800/4801) is off by default. Face and fingerprint still get logged — the Biometric service writes its own log regardless of audit policy — but a PIN or password used to unlock the lock screen leaves no trace at all until you turn this on.

One command fixes it (admin):

# Enable unlock auditing so PIN/password unlocks get recorded going forward
auditpol /set /subcategory:"Other Logon/Logoff Events" /success:enable
Note: This is retroactive to nothing — it only captures unlocks from the moment you enable it. On a managed device the audit policy is usually pushed by GPO/Intune anyway, so for a fleet you set this centrally (that's a topic for the next post).

Clue 3 — The face mystery: why face kept showing up as PIN

With auditing on, I locked the screen and deliberately signed in with password, then PIN, then face. The script reported: PIN, Password, Password… and the face unlock came back as PIN. Face was being misread.

So I dumped every event in the 3-minute window — Security, Biometrics and HelloForBusiness side by side. This is the line-up that cracked it:

Windows PowerShell — event timeline around one "face" unlock
Time Log Id What it means -------- --------- ---- ---------------------------------------- 16:13:09 Biometric 1702 "successfully unprotected data for user" 16:13:09 Biometric 1605 "secure component successfully authorized" 16:13:09 Hello 5702 Hello key protector written 16:13:09 Security 4801 workstation unlocked (no 1004 'identified' event anywhere)

There it was. A face unlock produces Event 1702 ("unprotected data") and 1605 ("authorized") — but no 1004. Fingerprint raises 1004; face on this hardware does not. My script only looked for 1004, so face fell through… and because face also tickles the Hello key (5702), it landed in the PIN bucket.

Compare a PIN-only unlock — same Hello key event, but the Biometric service stays silent:

Windows PowerShell — a PIN unlock vs a Password unlock
16:11:17 Hello 5702 Hello key used → PIN 16:11:17 Security 4801 workstation unlocked 16:11:41 Security 4801 workstation unlocked → Password (no Hello key, no biometric — nothing but the unlock)

That's the whole puzzle solved. Each method leaves a distinct fingerprint across the three logs.

The classification rules (validated on real hardware)

For every anchor (a 4624 logon or a 4801 unlock), look at what fired within a tight window (a few seconds — not 90):

MethodSignal at the moment of sign-in
FingerprintBiometrics 1004 "identified" (names the fingerprint sensor)
FaceBiometrics 1702 "unprotected data" with no 1004
PINHelloForBusiness 5702/5205 key use, but no biometric
PasswordAn unlock/logon with neither biometric nor Hello key
Tip: Order matters. Check 1004 (fingerprint) first, then 1702 (face), then the Hello key (PIN), then fall through to password. Fingerprint also raises 1702, so if you check 1702 first you'll mislabel fingerprint as face. The periodic 1608/1019 events are just the sensor idling — ignore them entirely.

How to verify it yourself

Before trusting any script, confirm the raw signals on your own device. Run this as administrator — it shows today's biometric hits straight from the source:

# Every successful biometric sign-in today, with the sensor name
Get-WinEvent -FilterHashtable @{
    LogName='Microsoft-Windows-Biometrics/Operational'; Id=1004;
    StartTime=(Get-Date).Date
} | Select-Object TimeCreated, Message
# Today's unlocks (needs the audit setting enabled, run as admin)
Get-WinEvent -FilterHashtable @{LogName='Security'; Id=4801; StartTime=(Get-Date).Date}
Note: These logs are admin-protected. Run PowerShell elevated. The full script below also needs admin because it reads the Security log — the only place password sign-ins are recorded.

The script: Get-TodayLogons.ps1

This is the heart of it — the classifier, applied to every sign-in and unlock today. It anchors on the Security log, classifies each anchor against the biometric and Hello signals, and also adds biometric sign-ins that happened before unlock auditing was on (so your fingerprint history isn't lost).

# Get-TodayLogons.ps1  (READ-ONLY, run as Administrator)
# Fingerprint=1004 | Face=1702 w/o 1004 | PIN=Hello key only | Password=nothing

$sid   = ([Security.Principal.WindowsIdentity]::GetCurrent()).User.Value
$start = (Get-Date).Date

# --- the three signal sources ---
$fp = Get-WinEvent @{LogName='...Biometrics/Operational'; Id=1004; StartTime=$start} |
      % { [pscustomobject]@{ Time=$_.TimeCreated; IsFinger=($_.Message -match 'fingerprint|MOC') } }
$faceAuth = (Get-WinEvent @{LogName='...Biometrics/Operational'; Id=1702; StartTime=$start}).TimeCreated
$hello    = (Get-WinEvent @{LogName='...HelloForBusiness/Operational'; Id=5205,5702; StartTime=$start}).TimeCreated

# --- anchors: real unlock/logon moments from the Security log ---
$anchors = Get-WinEvent @{LogName='Security'; Id=4624,4801; StartTime=$start} | ... (filter to this user)

function Near($t,$set,$sec){ @($set | ? { [math]::Abs(($t-$_).TotalSeconds) -le $sec }).Count -gt 0 }

# --- classify each sign-in / unlock ---
foreach($a in $anchors){
    $t = $a.Time
    $f = $fp | ? { [math]::Abs(($t-$_.Time).TotalSeconds) -le 8 } | select -First 1
    $method = if($f){ if($f.IsFinger){'Fingerprint'}else{'Face'} }
              elseif(Near $t $faceAuth 5){ 'Face' }
              elseif(Near $t $hello    8){ 'PIN' }
              else{ 'Password' }
    # ...emit { Date; Time; Method }
}
Tip: The -le 8 and -le 5 second windows are the fix for the original "everything is PIN" bug. A PIN unlock fires its Hello key event at the same second as the unlock; the background noise is always tens of seconds away. Keep the window tight.

Proof it worked

Here's the output after locking and unlocking with every method during testing. Fingerprint history from earlier in the day, then the deliberate face/PIN/password tests — all four, correctly identified, with the exact time and date:

PS C:\> .\Get-TodayLogons.ps1
=== Sign-ins (TODAY) === Date Time Method Source ---- ---- ------ ------ 2026-06-29 04:33:08 Fingerprint Logon (Security log) 2026-06-29 08:00:44 Fingerprint Biometric (sign-in/unlock) 2026-06-29 10:06:27 Fingerprint Biometric (sign-in/unlock) 2026-06-29 14:07:00 Fingerprint Biometric (sign-in/unlock) 2026-06-29 15:49:34 Fingerprint Biometric (sign-in/unlock) 2026-06-29 16:10:51 Face Unlock (Security log) 2026-06-29 16:10:58 Face Unlock (Security log) 2026-06-29 16:11:17 PIN Unlock (Security log) 2026-06-29 16:11:41 Password Unlock (Security log) 2026-06-29 16:12:34 Password Unlock (Security log) 2026-06-29 16:13:09 Face Unlock (Security log) 2026-06-29 16:22:44 Face Unlock (Security log)

That is the report Microsoft doesn't give you: face, fingerprint, PIN and password, per sign-in, with timestamps — generated entirely from data already sitting on the device.

Watch out: Two requirements for the password/PIN rows to be complete — (1) run as administrator (the Security log is admin-only), and (2) unlock auditing must be enabled. Face and fingerprint are always captured because the Biometric service logs them independently.
